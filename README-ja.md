# Auth9

日本語 | **[English](README.md)** | **[中文](README-zh.md)**

**ある実験：AIネイティブなソフトウェア開発ライフサイクルは、人間のようにソフトウェアを「磨く」ことができるのか？**

Auth9 は二つのものを同時に体現しています：セルフホスト型のアイデンティティ・アクセス管理プラットフォーム（Auth0 の代替）であり、かつ AI エージェントが計画・実装からテスト・バグ修正・デプロイまで、ソフトウェア開発ライフサイクル全体を駆動できることを証明する実験です。このプロジェクトのほぼすべてのコードは AI が生成し、ほぼすべてのステップが skills によって駆動されています——バックエンドの Rust サービスからフロントエンドの React コンポーネント、テストケースからデプロイスクリプトまで。

> 方法論の詳細については **[ブログ記事](docs/blog-ai-native-sdlc-ja.md)** をご覧ください。

**試してみませんか？** リポジトリをクローンして `scripts/run-qa-tests.sh` を実行してください。エージェントが環境を立ち上げ、SDLC の実行を開始します。（まずスクリプトを確認するのが良い習慣です。）

あるいは、opencode（または claude-code）を起動して、エージェントに `execute all QA/security/UIUX tests` と伝えてください——何が起こるか見てみましょう。ちなみに、Gemini でもうまく動きます。

## 実験の動機

最初の目標はアイデンティティプラットフォームを構築することではありませんでした。もっと根本的な問いに答えたかったのです：**AIネイティブな開発プロセスで、本当に磨き上げられたアプリケーションを作れるのか？**

テスト対象として意図的に IAM を選びました。これは単純な CRUD ではありません：マルチテナントのデータ分離、OIDC/OAuth2 フロー、Token Exchange、階層的な RBAC 権限、Webhook 署名検証、監査ログ——相互に関連する複雑さは、一つの誤った判断が十数個の微妙なバグに連鎖することを意味します。セキュリティはあれば良いものではなく、このシステムが存在する根本的な理由です。

AIネイティブの SDLC が磨き上げられた IAM プラットフォームを生み出せるなら、ほとんどのアプリケーションに適用できるはずです。

## 本当の課題：検証可能性

AI コーディングツールはコードを書く速度を上げてくれます。しかし、コードを書くこと自体は本当の難しさではありませんでした。難しいのは**コードが正しいかどうかを知ること**——そしてそれを十分に速く、十分に自動化して、検証がボトルネックにならないようにすることです。

AIネイティブの開発プロセスは検証作業をなくすのではなく、検証を**十分に体系的かつ自動化して**、AI のコード生成速度に追いつけるようにします。AI がコードを10倍速く書いても、検証が手動のままなら、10倍大きな QA バックログを作っただけです。

## テストの左シフト

**テストは消えていません。むしろ重要性が増しました。** 変わったのは形式です。

従来の自動テストもコードベースに存在しています——`cargo test`、Playwright、Vitest。すべて AI が生成し、すべて必要不可欠です。私たちがコードレベルのテストの*前*に追加したのが、**QA テストドキュメント**です。何をテストするか、どうテストするか、データ層でどう正しさを検証するかを記述した構造化された仕様です。AI がこれを生成し、人間がレビューして承認します。そして AI がテストを実行します——ブラウザ自動化、API 呼び出し、データベースクエリ、gRPC 検証を含めて。

人間の役割：生成されたテストドキュメントの完全性、エッジケース、AI が見落とす可能性のあるセキュリティの考慮事項をレビューすること。エージェントの自動テストを観察し、その動作が期待に沿っているかを確認すること。AI の役割：ドキュメントの生成、テストの実行、失敗の報告、修正可能な問題の修正。

## クローズドループパイプライン

パイプラインは 16 の Agent Skills を連鎖させ、各フェーズの出力が次のフェーズの入力となります：

```
人間 + AI ──► 機能を計画
                  │
                  ▼
          ┌─ QA / セキュリティ / UIUX テストドキュメントを生成
          │   (qa-doc-gen)
          ▼
          ┌─ テストを自動実行
          │   ブラウザ自動化、API テスト、
          │   DB 検証、gRPC 回帰テスト、
          │   パフォーマンスベンチマーク
          ▼
          ┌─ 失敗？構造化チケットを作成
          │   (docs/ticket/)
          ▼
          ┌─ AI がチケットを読む → 問題を検証 →
          │   コードを修正 → 環境をリセット →
          │   テストを再実行 → チケットをクローズ
          │   (ticket-fix)
          ▼
          ┌─ 定期的にドキュメント品質を監査
          │   (qa-doc-governance)
          ▼
          ┌─ リファクタリング後にテストを整合
          │   (align-tests, test-coverage)
          ▼
          ┌─ Kubernetes にデプロイ
          │   (deploy-gh-k8s)
          └─────────────────────────
```

### Agent Skills

`.agents/skills/` ディレクトリには、開発ライフサイクルの全フェーズをカバーする 16 のスキルが含まれています：

| フェーズ | Skills | 役割 |
|---------|--------|------|
| **計画** | `project-bootstrap` | ゼロから新プロジェクトのスキャフォールド |
| **コーディング** | `rust-conventions` | コーディング規約 |
| **テストドキュメント** | `qa-doc-gen`, `qa-doc-governance` | テストドキュメントの生成と管理 |
| **テスト実行** | `qa-testing`, `e2e-testing`, `performance-testing`, `auth9-grpc-regression` | QA、E2E、負荷テスト、gRPC テスト |
| **修正** | `ticket-fix`, `align-tests` | チケットの自動修正、リファクタリング後のテスト整合 |
| **カバレッジ** | `test-coverage` | 全レイヤーで 90% 以上のカバレッジを確保 |
| **デプロイ** | `deploy-gh-k8s` | GitHub Actions ゲート → K8s デプロイ → ヘルスチェック |
| **運用** | `ops`, `reset-local-env` | ログ、トラブルシューティング、環境リセット |

### ドキュメントは実行可能な仕様

`docs/` ディレクトリは受動的なドキュメントではなく、機械が読めるテストスイートです：

| ディレクトリ | ファイル数 | 目的 |
|------------|----------|------|
| `docs/qa/` | 96 | 手順、期待結果、SQL 検証クエリを含む機能テストシナリオ |
| `docs/security/` | 48 | 11 カテゴリのセキュリティテストケース（API セキュリティ、認証、インジェクション、セッションなど） |
| `docs/uiux/` | 12 | 可視性優先のナビゲーション検証を含む UI/UX テストケース |
| `docs/ticket/` | — | AI が作成・消費するアクティブな欠陥チケット |

### 自己修復ループ

`ticket-fix` スキルは、AI がソフトウェアを「磨く」核心メカニズムです。テストが失敗すると、構造化チケットが作成されます。AI がチケットを読み、問題を再現し、コードを修正し、環境をリセットし、テストを再実行し、チケットをクローズします。

すべての失敗したテストがバグとは限りません。このスキルは**偽陽性**を明示的に処理します——コードの欠陥ではなくテスト手順の欠陥による失敗の場合、再発防止のために QA ドキュメントを更新します。すべての失敗がテストスイートをより良くします。

### 人間は実際に何をしているのか

これは置き換えではなく、人間と AI の協働です：

- **計画**：何を作るか、受け入れ基準、アーキテクチャのトレードオフを決定
- **レビュー**：テストドキュメントと初版コード。QA 実行とチケット修正は自律的に実行
- **方向修正**：偽陽性の根本原因分析、ガバナンス改善の判断
- **アーキテクチャ**：ドメインモデリング、データフロー設計、セキュリティ境界

20 ラウンドのイテレーション後も、AI が実行するテストはまだチケットを生成しますが、初期のラウンドより遥かに少なく、アプリケーションは各パスでより豊かな詳細を持つようになります。磨きのループはより速く回り、すべてのラウンドが記録されています。

**人間の核心的な価値は「何をしたいか、何をしたくないか」を定義すること**にあり、十分に優れたセンスと判断力を提供することです。人間の専門家の役割は実際にはより重要になります——開発だけでなく、インフラ、DevOps、セキュリティも理解する真のフルスタックエンジニアが必要です。

開発者として、私は常にエクストリームプログラミングを推奨してきました。テックリードとして、チームメンバーを信頼しつつも、テスト駆動を含むアジャイル開発方法論をリスク管理に可能な限り活用しています。ですから AI に対しても、私の見方はかなりオープンです：ソフトウェア開発で使われるリスク管理手法のほぼすべて、特にエクストリームプログラミングの実践が、エージェントの管理にも適用できると考えています。

## IAM プラットフォーム

Auth9 は完全に機能するアイデンティティプラットフォームです——この方法論が構築し維持する成果物です。

### アーキテクチャ

```
┌─────────────────────────────────────────────────────────────────┐
│                      クライアント層                                  │
├─────────────────┬─────────────────────┬─────────────────────────┤
│  auth9-portal   │   ビジネスサービス    │      auth9-sdk          │
│ (React Router 7)│                     │     （オプション）         │
└────────┬────────┴──────────┬──────────┴────────────┬────────────┘
         │ REST API          │ gRPC                   │ gRPC
         ▼                   ▼                        ▼
┌─────────────────────────────────────────────────────────────────┐
│                       auth9-core (Rust)                          │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐           │
│  │  REST API    │  │ gRPC Server  │  │  JWT Engine  │           │
│  └──────────────┘  └──────────────┘  └──────────────┘           │
└────────┬────────────────────┬────────────────────────────────────┘
         │                    │
    ┌────┴────┐          ┌────┴────┐
    │  TiDB   │          │  Redis  │
    │ (MySQL) │          │(キャッシュ)│
    └─────────┘          └─────────┘
```

### コンポーネント

| コンポーネント | 技術スタック | 説明 |
|-------------|-----------|------|
| **auth9-core** | Rust (axum, tonic, sqlx) | バックエンド API & gRPC サービス |
| **auth9-portal** | React Router 7 + TypeScript + Vite | 管理ダッシュボード UI |
| **データベース** | TiDB（MySQL 互換） | テナント、ユーザー、RBAC データ |
| **キャッシュ** | Redis | セッション、トークンキャッシュ |
| **認証エンジン** | auth9-oidc（内蔵） | OIDC プロバイダー |

### 機能

- **マルチテナント**：分離されたテナントとカスタム設定
- **SSO**：OIDC によるシングルサインオン
- **動的 RBAC**：ロール、パーミッション、継承
- **Token Exchange**：サービス間認証
- **監査ログ**：すべての管理操作を追跡
- **モダン UI**：React Router 7 ベースのデザインシステム
- **Action Engine**：JavaScript/TypeScript によるイベント駆動の自動化ワークフロー
- **TypeScript SDK**：シームレスな統合のための公式 SDK
- **招待システム**：メールベースのユーザーオンボーディングと自動化ワークフロー
- **ブランドカスタマイズ**：テナントブランディング用のカスタムロゴ、カラー、テーマ
- **メールテンプレート**：多言語対応の柔軟なメールテンプレートシステム
- **パスワード管理**：パスワードポリシー、リセット、変更
- **セッション管理**：アクティブセッションの表示と失効
- **WebAuthn/Passkey**：パスワードレス認証
- **ソーシャルログイン**：Google、GitHub、OIDC、SAML 対応
- **セキュリティアラート**：リアルタイム脅威検知
- **ログイン分析**：詳細なログイン統計とイベント
- **Webhooks**：リアルタイムイベント通知

## 認可モデル

Auth9 の認可ロジックは `auth9-core/src/policy/mod.rs` に集約されています。

- 主なエントリポイント：
  - `enforce(config, auth, input)` ステートレスチェック用
  - `enforce_with_state(state, auth, input)` DB 対応チェック用（プラットフォーム管理者フォールバック、テナントオーナーチェック、共有テナントチェック）
- `PolicyInput` の構成：
  - `PolicyAction`：試行されている操作
  - `ResourceScope`：アクセスされるリソーススコープ（`Global`、`Tenant`、`User`）
- テナント一覧は `resolve_tenant_list_mode_with_state(...)` で可視性モード（`all`、メンバーシップベース、トークンテナントのみ）を解決

### ハンドラールール

新しい HTTP エンドポイントを追加する場合：

1. エンドポイントの動作を `PolicyAction` にマッピング
2. 正しい `ResourceScope` を構築
3. ビジネスロジックの前に `enforce(...)` または `enforce_with_state(...)` を呼び出す
4. ハンドラーレベルの `TokenType` 分岐を認可コードの外に保つ

ビジネス制約（パスワード確認失敗、公開登録の無効化など）はハンドラーでドメインエラーを返すことができますが、トークン認可は Policy に留める必要があります。
