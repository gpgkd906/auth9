# ある実験：AIネイティブなソフトウェア開発ライフサイクルは、人間のようにソフトウェアを「磨く」ことができるのか？

*もう一度、テストの左シフト*

---

## 実験の動機

最初の目標はアイデンティティプラットフォームを構築することではありませんでした。もっと根本的な問いに答えたかったのです：**AIネイティブな開発プロセスで、本当に磨き上げられたアプリケーションを作れるのか？**

おもちゃのデモでも、週末プロジェクトでもなく、本物のシステム——方法論を検証するのに十分な複雑さを持ち、セキュリティ要件が十分に高く、「まあまあ使える」程度では通用しないもの。

テスト対象として IAM（アイデンティティ・アクセス管理）を選びました。これは単純な CRUD ではありません：マルチテナントのデータ分離、OIDC/OAuth2 フロー、Token Exchange、階層的な RBAC 権限、Webhook 署名検証、監査ログ——相互に関連する複雑さは、一つの誤った判断が十数個の微妙なバグに連鎖することを意味します。ここではセキュリティはあれば良いものではなく、システムが存在する根本的な理由です。

最終成果は [Auth9](https://github.com/gpgkd906/auth9) です。Auth0 のセルフホスト型代替として、Rust、React Router 7、TiDB、Keycloak で構築されています。このプロジェクトのほぼすべてのコードは AI が生成し、ほぼすべてのステップが skills によって駆動されています——バックエンドの Rust サービスからフロントエンドの React コンポーネント、テストケースからデプロイスクリプトまで。しかし Auth9 は単なる成果物です。実験の真の主役はこの開発パイプライン自体です。

## 本当の課題：検証可能性

AI コーディングツールは確かにコードを書く速度を上げてくれます。GitHub Copilot、Cursor、Claude Code——どれもその約束を果たしています。

しかし、コードを書くこと自体は本当の難しさではありませんでした。難しいのは**コードが正しいかどうかを知ること**——そしてそれを十分に速く、十分に自動化して、検証がボトルネックにならないようにすることです。

AI は大量のコードを書き、見た目は良いのですが、テストした瞬間に失敗するかもしれません——依存関係の欠落、lint エラーなど様々です。AI がミスするたびにプロンプトを出すのは面倒ですが、大丈夫です——AI にコードを書き終えたら自分でテストさせれば良いのです。

AIネイティブの開発プロセスは検証作業をなくすのではなく、検証を**十分に体系的かつ自動化して**、AI のコード生成速度に追いつけるようにします。AI がコードを10倍速く書いても、検証が手動のままなら、10倍大きな QA バックログを作っただけです。

## テストの左シフト

はっきり言います：**テストは消えていません。むしろ重要性が増しました。** 変わったのは形式です。

従来の自動テスト——ユニットテスト、統合テスト、エンドツーエンドテスト——はコードベースに依然として存在しています。`cargo test` は外部依存なしで高速に実行されます。Playwright がエンドツーエンドフローを駆動します。Vitest がフロントエンドをカバーします。これらの手法は依然として有効で不可欠です。もちろん、すべて AI が生成したものです。

私たちがコードレベルのテストの*前*に追加したのが、**QA テストドキュメント**です。何をテストするか、どうテストするか、データ層でどう正しさを検証するかを記述した構造化された仕様です。AI がこれを生成し、人間がレビューして承認します。そして AI がテストを実行します——ブラウザ自動化、API 呼び出し、データベースクエリ、gRPC 検証を含めて。本質的には従来の手動テストに近いですが、自動化されています。もちろん、まだ完全に手動テストを置き換えることはできません。

これがドキュメントへのテスト左シフトです。テスト計画は機能がリリースされた後に走り書きするものではなくなりました。機能が計画された後に最初に生成される成果物であり、その後のすべてを駆動します：どんなコードを書くか、何を検証するか、何かが壊れたときに何を捕捉するか。

人間の役割：生成されたテストドキュメントの完全性、エッジケース、AI が見落とす可能性のあるセキュリティの考慮事項をレビューすること。エージェントの自動テストを観察し、テスト動作が期待に沿っているか、あるいは不正がないかを確認すること。AI の役割：ドキュメントの生成、テストの実行、失敗の報告、修正可能な問題の修正。そして QA テストドキュメント自体がユーザーストーリーを反映しています。

仕様駆動開発についての大胆な意見：**QA テストドキュメントが仕様そのものです。** 今、デザインドキュメントを仕様として AI 開発を駆動することへの期待が高まっています。しかしデザインドキュメントが記述するのは*何が欲しいか*であり、それを得られたかどうかは教えてくれません。QA テストドキュメントは両方を兼ね備えています：期待される動作を定義し、*同時に*稼働中のシステムに対して自動的に検証できます。デザインドキュメントは静かに腐敗しますが、QA ドキュメントは現実が仕様から逸脱した瞬間に大きな声で失敗を知らせます。仕様が実行可能でないなら、それは単なる願望です。

## クローズドループパイプライン

パイプラインは 16 の Agent Skills を連鎖させ、各フェーズの出力が次のフェーズの入力となります：

```
人間 + AI ──► 機能を計画
                  │
                  ▼
          ┌─ QA / セキュリティ / UIUX テストドキュメントを生成
          │   (qa-doc-gen)
          ▼
          ┌─ テストを自動実行
          │   ブラウザ自動化、API テスト、
          │   DB 検証、gRPC 回帰テスト、
          │   パフォーマンスベンチマーク
          ▼
          ┌─ 失敗？構造化チケットを作成
          │   (docs/ticket/)
          ▼
          ┌─ AI がチケットを読む → 問題を検証 →
          │   コードを修正 → 環境をリセット →
          │   テストを再実行 → チケットをクローズ
          │   (ticket-fix)
          ▼
          ┌─ 定期的にドキュメント品質を監査
          │   (qa-doc-governance)
          ▼
          ┌─ リファクタリング後にテストを整合
          │   (align-tests, test-coverage)
          ▼
          ┌─ Kubernetes にデプロイ
          │   (deploy-gh-k8s)
          └─────────────────────────
```

各スキルは独立した Agent Skill 定義です——指示、ワークフローステップ、ツールスクリプトを含む Markdown ファイルです：

| フェーズ | Skills | 役割 |
|---------|--------|------|
| **計画** | `project-bootstrap` | ゼロから新プロジェクトのスキャフォールド |
| **コーディング** | `rust-conventions`, `keycloak-theme` | コーディング規約、テーマ開発 |
| **テストドキュメント** | `qa-doc-gen`, `qa-doc-governance` | テストドキュメントの生成と管理 |
| **テスト実行** | `qa-testing`, `e2e-testing`, `performance-testing`, `auth9-grpc-regression` | QA、E2E、負荷テスト、gRPC テスト |
| **修正** | `ticket-fix`, `align-tests` | チケットの自動修正、リファクタリング後のテスト整合 |
| **カバレッジ** | `test-coverage` | 全レイヤーで 90% 以上のカバレッジを確保 |
| **デプロイ** | `deploy-gh-k8s` | GitHub Actions ゲート → K8s デプロイ → ヘルスチェック |
| **運用** | `ops`, `reset-local-env` | ログ、トラブルシューティング、環境リセット |

重要な洞察：**Agent Skills は組み合わせて使えます**。機能が計画された後、`qa-doc-gen` がテストドキュメントを生成し、`qa-testing` がそれを実行し、失敗は `docs/ticket/` のチケットになり、`ticket-fix` がチケットを読んでコードを修正し、環境をリセットし、テストを再実行し、チケットをクローズします。`qa-doc-governance` が定期的にすべてを監査し、ドキュメントの腐敗を防ぎます。

## ドキュメントは実行可能な仕様

この仕組みが機能するのは、テストドキュメントが人間と AI の両方のために構造化されているからです。テナント CRUD テストスイートからの実際のシナリオを示します：

> ### シナリオ 1：テナント管理の入口可視性とテナント作成
>
> **初期状態**
> - ユーザーは管理ダッシュボードにログイン済み
> - 同じ名前またはスラグのテナントがデータベースに存在しない
>
> **テスト手順**
> 1. 左サイドバーに「テナント管理」メニューエントリが存在することを確認
> 2. 「テナント管理」をクリックしてテナント一覧に入る
> 3. 「テナント作成」ボタンをクリック
> 4. フォームに入力：
>    - 名前：`Test Company`
>    - スラグ：`test-company`
>    - Logo URL：`https://example.com/logo.png`
> 5. 「作成」ボタンをクリック
>
> **期待結果**
> - 成功通知が表示される
> - サイドバーエントリが表示され、クリック可能
> - テナントが一覧に「Active」ステータスで表示される
>
> **期待されるデータ状態**
> ```sql
> SELECT id, name, slug, logo_url, status FROM tenants WHERE slug = 'test-company';
> -- 期待: 1レコードが存在、status = 'active'
>
> SELECT action, resource_type FROM audit_logs WHERE resource_type = 'tenant' ORDER BY created_at DESC LIMIT 1;
> -- 期待: action = 'tenant.create'
> ```

SQL 検証が重要なポイントです。AI は UI が「成功」と表示するかどうかだけをチェックするのではなく、データベースにクエリを投げてデータが実際に正しく格納されたことを確認します。これにより、フロントエンドは成功を表示するがバックエンドがサイレントに失敗するという一連のバグを捕捉できます。

このようなドキュメントが **156 件**あります：96 の QA、48 のセキュリティ、12 の UI/UX——それぞれが AI が生成し人間がレビューした実行可能な仕様です。

情報交換の媒体としてファイルシステムを選びました（例：`docs/qa`、`docs/ticket` ディレクトリ）。その理由は単純です——エージェントは bash 操作に優れており、ファイルシステムとのやり取りは自然で正確かつ効率的です。このファイルベースのワークフローにより、検証プロセス全体が透明で追跡可能になります。

## 自己修復ループ：AI がソフトウェアを「磨く」方法

`ticket-fix` スキルはパイプラインの最も興味深い部分であり、AI がソフトウェアを「磨く」核心メカニズムです。テストが失敗すると、構造化チケットが作成されます。そして：

1. **チケットの読み取りと検証** — シナリオ、手順、期待結果と実際の結果、環境の詳細、SQL チェックを解析。
2. **再現** — 現在の実装に対して失敗するテストを実行。
3. **必要に応じて修正** — チケットのスコープに限定した最小限のコード修正を実装。
4. **環境リセット** — 必ず実行。`./scripts/reset-docker.sh` でクリーンな状態を取得。
5. **QA ステップの再実行** — チケットの正確な手順と SQL 検証に従って実行。エビデンスを取得。
6. **偽陽性の分析** — これが仕組みを機能させるニュアンスです。
7. **チケットのクローズ** — チケットファイルを削除し、結果をサマリー。

### 偽陽性分析

すべての失敗したテストがバグとは限りません。`ticket-fix` スキルは偽陽性を明示的に処理します——コードの欠陥ではなくテスト手順の欠陥による失敗：

- テストコマンドに必要な認証ヘッダーが欠落（例：HMAC 署名）
- 前提条件が不完全または曖昧（「webhook が存在する必要がある」が作成手順なし）
- 環境の前提が Docker のデフォルト設定と不一致
- テストデータが存在しないエンティティを参照し、フォールバック処理なし

偽陽性が確認されると、スキルはチケットをクローズするだけでなく、同じ偽陽性の再発を防ぐために **QA ドキュメントを更新**します。暗黙の要件を明示的にし、サンプルコマンドがコピー＆ペーストですぐ使えるようにし、一般的な失敗パターンのトラブルシューティングテーブルを追加します。

これは、テストが失敗するたびに、それが本当のバグであろうとテスト自体の問題であろうと、テストスイートが*より良くなる*ことを意味します。これが「磨き」の本質です——繰り返しの検証と修正を通じて、ソフトウェアとテストドキュメントの両方がより洗練されていきます。

## ドキュメントガバナンス

テストドキュメントは腐敗します。ルートが変わり、API が進化し、パーミッションが再構築され——突然、テストスイートの半分がもう存在しない動作をテストしていることになります。

すべての QA テストドキュメントはエージェントによって繰り返し実行されるため、ドキュメントが古くなると、大量の偽陽性チケットレポートに気づきます。ドキュメントを最新に保つ最良の戦略は、使い続けることです。

`qa-doc-governance` スキルは重要度分類付きの定期監査を実行します：

- **P0**：ナビゲーション不通またはテストフローが使用不可（チェックリスト欠落、誤解を招く認証手順、インデックスの偏り）
- **P1**：ガバナンスの逸脱（1ファイル5シナリオ超、UI 関連ドキュメントに入口可視性の説明欠落）
- **P2**：スタイルの一貫性（命名の整合、文言の標準化）

ワークフロー：全ドキュメントの lint → 重要度分類 → 優先度順に修正 → インデックスとマニフェストの同期 → 検証とレポート。新機能やリファクタリングの後に実行でき、定期的（例：週次）、メジャーリリース前、インシデント後にも実行可能です。

一つの厳格なルール：**すべてのドキュメントに回帰テストチェックリストを含める必要があります**。例外なし。ドキュメントの他の部分が逸脱しても、チェックリストが最低限の検証パスを提供します。

## クロスドキュメント影響分析

機能が変更されると、影響は一つのテストドキュメントだけにとどまりません。`qa-doc-gen` スキルには、全 156 ドキュメントをスキャンする必須のクロスドキュメント影響分析が含まれています：

- ルート変更（`/dashboard`、`/tenant/select`、`/auth/callback`）
- トークンモデル変更（`id_token`、`tenant token`、`token exchange`）
- パーミッション/認証変更（`401`、`403`、`scope`、`audience`、`role`）
- UI ナビゲーションテキストと期待されるリダイレクトパス

影響を受ける各ドキュメントに対して、影響を分類します：

1. **パッチ必要** — 手順、期待結果、またはセキュリティアサーションを更新
2. **注記必要** — 前提条件や分岐パスの注記を追加し、テスターの混乱を防ぐ
3. **変更不要** — 影響を受けない理由を明示的に記録

重要なルール：**古いドキュメントが陳腐化しているときに、新しいドキュメントだけを作成して終わりにしてはいけません。**

## 人間は実際に何をしているのか

これは置き換えではなく、人間と AI の協働です。日々の私の実際の作業内容：

- **計画**：何を作るか決定し、受け入れ基準を定義し、アーキテクチャのトレードオフを選択。AI がスキャフォールドし、私が方向を決めます。
- **レビュー**：生成されたテストドキュメントと、新機能/リファクタリングの初版コードをレビュー。QA 実行とチケット修正のサイクルは AI が自律的に実行します。レビューの労力は最も重要なところに集中します——正しさを定義する*仕様*と、方向を設定する*初期実装*。
- **方向修正**：パイプラインが偽陽性を生成した場合、根本原因分析をレビュー。ガバナンスが P0 をフラグした場合、修正アプローチを決定。
- **アーキテクチャ**：ドメインモデリング、データフロー設計、セキュリティ境界——これらはコードだけでなく*システム*についての判断力が必要です。

このプロジェクトでは、同じ量の個人的な時間を投資して、10倍以上のアウトプットを達成しました。

これは AI にすべてを監視なしで任せられるという意味ではありません。20 ラウンドのイテレーション後も、AI が実行するテストはまだチケットを生成します——しかし初期のラウンドより遥かに少なく、アプリケーションは各パスでより豊かな詳細を持つようになります。この体験は人間のエンジニアがすることと驚くほど似ています：繰り返しの QA を通じてバグを層ごとに排除し、ソフトウェアを磨いていきます。違いは、磨きのループがより速く回るようになり、すべてのラウンドが記録されていることです。

**人間の核心的な価値は「何をしたいか、何をしたくないか」を定義すること**にあり、十分に優れたセンスと判断力を提供することです。AI はこの定義を効率的に実装し、自動テストを通じて継続的に検証し修正する責任を負います。

重要な洞察：人間の専門家の役割は実際にはより重要になります——エージェントが検証を実行できる環境を構築する必要があり、そのためには人間の専門家が開発だけでなくインフラにも精通している必要があります。さらに、DevOps やセキュリティの課題への取り組みが、人間の専門家がドキュメントをより包括的にレビューし、このプロセスを継続的に改善するのに役立ちます。つまり、真のフルスタックエンジニアが必要なのです。

開発者として、私は常にエクストリームプログラミングを推奨してきました。テックリードとして、チームメンバーを信頼しつつも、テスト駆動を含むアジャイル開発方法論をリスク管理に可能な限り活用しています。ですから AI に対しても、私の見方はかなりオープンです：ソフトウェア開発で使われるリスク管理手法のほぼすべて、特にエクストリームプログラミングの実践が、エージェントの管理にも適用できると考えています。

## 正直な限界

このアプローチは魔法ではありません。まだうまくいかないことを挙げます：

- **全く新しいアーキテクチャ判断** — AI は以前見たことのあるパターンの実装には優れています。真に新しい設計選択については、人間のアーキテクトがリードします。
- **セキュリティレビュー** — AI はセキュリティテストドキュメントを生成し実行しますが、脅威モデル自体には人間のセキュリティ専門知識が必要です。自動テストは既知のパターンを捕捉しますが、新しい攻撃ベクトルを発見するものではありません。
- **プロセスにはプロジェクト適応が必要** — プロセス自体が特定のプロジェクトに合わせて最適化が必要です。例えば、UI テストが不要なプロジェクトもあれば、API やデータ層の検証に重点を置くプロジェクトもあります。スキルの組み合わせやテストの重点は、プロジェクトの特性に応じて調整が必要です。

## 数字で見る

- **16** の Agent Skills が開発ライフサイクル全体をカバー
- **156** のテストドキュメント（96 QA + 48 セキュリティ + 12 UI/UX）
- **9** のツールスクリプト（トークン生成、API テスト、gRPC スモークテスト）
- **約 2,300** 行のスキル定義
- **1** 人の人間

## なぜ IAM なのか？

なぜ開発方法論をテストするためにアイデンティティプラットフォームを構築するのか？

簡単な問題は方法論を検証できないからです。TODO アプリやブログエンジンでは何も証明できません——ドメインが簡単すぎて、どんなアプローチでもうまくいくように見えます。IAM は以下に向き合うことを強制します：

- **ごまかせないセキュリティ** — トークン検証、パーミッション適用、インジェクション防御。48 のセキュリティテストドキュメントが存在するのは、認証の領域では「自分のマシンでは動いた」が許されないからです。
- **マルチシステム連携** — Keycloak、TiDB、Redis、Rust バックエンド、React フロントエンド。Token Exchange フローのバグはすべてのレイヤーに影響します。
- **状態の複雑さ** — マルチテナントのデータ分離、階層的 RBAC、セッション管理。状態空間が大きすぎて手動テストではカバーできません。

AIネイティブの SDLC が磨き上げられた IAM プラットフォームを生み出せるなら、ほとんどのアプリケーションに適用できるはずです。

## 自分で試してみる

試してみませんか？リポジトリをクローンして `scripts/run-qa-tests.sh` を実行してください。エージェントが環境を立ち上げ、SDLC の実行を開始します。もちろん、まずスクリプトを確認するのが良い習慣です。

あるいは、opencode（または claude-code）を起動して、エージェントに `execute all QA/security/UIUX tests` と伝えてください——何が起こるか見てみましょう。ちなみに、Gemini でもうまく動きます。

パイプライン全体がオープンソースです：[github.com/gpgkd906/auth9](https://github.com/gpgkd906/auth9)

この方法論はアイデンティティプラットフォーム専用ではありません。AI 主導の開発についての考え方です：**AI を使ってコードを速く書くだけでなく、計画・テスト・修正・デプロイのループを閉じるために使いましょう。**

*Auth9 はセルフホスト型のアイデンティティ・アクセス管理プラットフォーム（Auth0 の代替）で、Rust、React Router 7、TiDB、Keycloak で構築されています。AIネイティブの開発パイプラインは、16 のカスタムスキルを持つ Agent Skills アーキテクチャを使用しています。*
